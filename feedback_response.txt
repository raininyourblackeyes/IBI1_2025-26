During the first few practicals, I did not receive formative ICA feedback, so I asked my classmates to review my code instead. They helped me check whether the code was understandable and whether the output matched the task requirements. Their feedback was mainly about basic issues, such as adding more comments, using clearer variable names, and making the output easier to read. They also pointed out some small problems, for example, some steps were not explained clearly enough, and some parts of the code could be organized better.

In the later practicals, I received more feedback from my partner and classmates during the coding process. This feedback focused more on code readability, special cases, and possible errors. For example, they reminded me to consider unusual input, such as negative numbers, wrong input formats, or results that were biologically or mathematically impossible. In the SIR model, I also noticed that if probabilities or population values were not checked carefully, the results could become unrealistic, such as negative population numbers.

Based on this feedback, I made several improvements. I added more comments to explain important steps, changed some variable names to make the code easier to follow, and tested the code step by step. When problems appeared, I used print statements to check intermediate values and locate the error. Through these practicals, I learned that code should not only run successfully, but also be readable, reliable, and able to handle unexpected situations.
